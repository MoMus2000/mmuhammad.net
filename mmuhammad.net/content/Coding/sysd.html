Writing services with systemd ...
By Mustafa Muhammad
Systemd is an init system, that replaces SysVInit and upstart. It was created at red hat. Upon scouring the internet to prepare content for this write up, I've come to learn that it's also very polarizing to some sys-admins and developers.
However my write up is not related to the init system, nor the controversies regarding the use of systemd.
I simply find it to be a simple way to schedule my scripts and programs to run as services on an Ubuntu VM.
For me, the whole point of systemd is to be able to effectively manage services which are also referred to as units or daemons.

If you navigate to /etc/systemd/system (Ubuntu) or equivalent on your system, you will see a plethora of service files, these are what contain the information that is needed to be able to launch the processes as daemon when systemctl gets run.

Run these series of commands:
  • systemctl daemon-reload (to load go_server.service file)
  • systemctl enable go_server.service
  • systemctl start go_server.service
To check service logs we make use of a utility called journalctl.
I usually make use of this command here to get output from specific services.
In conclusion systemd is a very versatile service for software developers that helps to manage applications that require services to be up, running and constantly communicating with one another. It also allows the functionality to be able to restart on errors or create timer files that trigger services based on certain criteria (for e.g time), but more on that some other time.
